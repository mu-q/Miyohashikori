# 需求文档：Miyohashikori AI 桌宠

## 1. 项目概述

本项目是一个基于 Qt Widgets 的 Windows 桌面 AI 桌宠应用。角色原型为《甜糖热恋》中的圣代桥冰织，核心目标是提供轻量、常驻、具有陪伴感的桌面交互体验。

应用通过透明置顶窗口展示角色立绘，通过文本对话与用户交互，并根据 AI 回复自动切换角色表情。后续可扩展 TTS 播报、托盘控制、单实例保护和日志等能力。

## 2. 产品目标

### 2.1 核心目标

- 在桌面上持续显示一个轻量透明的角色窗口
- 支持用户与角色进行自然文本对话
- AI 回复需要尽可能稳定地符合“冰织”的角色设定
- AI 回复应附带 emotion 标签，驱动立绘表情变化

### 2.2 当前阶段目标

当前阶段优先实现 MVP：

- 配置加载与保存
- 基于 OpenAI 兼容接口的真实 LLM 对话
- 文本输入与回复展示
- emotion 标签解析
- 基础立绘表情切换

## 3. 技术方案约束

### 3.1 桌面端技术栈

- UI 框架：Qt Widgets
- 语言标准：C++17
- 网络：QNetworkAccessManager
- 配置存储：本地 JSON
- 语音播放：Qt Multimedia（后续阶段）

### 3.2 AI 接口约束

- LLM 接口类型：OpenAI 兼容 Chat Completions HTTP API
- 当前推荐模型：千问 `qwen-plus`
- 当前推荐服务：阿里云百炼 DashScope 兼容接口
- 返回结果必须能解析出 `[emotion:xxx]`

### 3.3 配置文件约束

- 配置文件路径：`~/.hyori/config.json`
- 日志路径：`~/.hyori/logs/`

## 4. 术语定义

- `DesktopPetWindow`：桌宠主窗口，透明、无边框、置顶
- `Sprite`：桌宠当前显示的立绘图片
- `Expression`：角色当前表情状态
- `SpeechBubble`：角色旁边显示的对话气泡
- `InputDialog`：用户输入文本的对话窗口
- `IAiSession`：AI 对话抽象接口
- `OpenAiChatSession`：基于 OpenAI 兼容接口的真实 AI 会话实现
- `ConfigManager`：配置加载、保存和管理组件
- `ChatHistory`：AI 上下文消息管理组件
- `EmotionParser`：emotion 标签解析组件

## 5. 功能需求

### 5.1 桌宠主窗口

#### 用户故事

作为用户，我希望在桌面上看到一个常驻的角色立绘窗口，以便获得持续的视觉陪伴感。

#### 验收标准

1. 应用启动后，系统应显示一个无边框、透明背景、始终置顶的桌宠窗口。
2. 窗口应展示带透明通道的 PNG 立绘。
3. 窗口应支持鼠标拖动移动位置。
4. 窗口位置应在关闭或拖动后保存，并在下次启动时恢复。
5. 右键点击桌宠时，应弹出上下文菜单。

### 5.2 文本对话

#### 用户故事

作为用户，我希望输入文字与冰织对话，以便获得陪伴与反馈。

#### 验收标准

1. 用户应能输入一段文本并提交给 AI。
2. 输入内容为空时，不应发送请求。
3. AI 回复应异步返回，不阻塞 UI。
4. 回复失败时，应向用户显示明确错误提示。
5. 当前阶段可先使用主窗口输入框，后续替换为独立 `InputDialog`。

### 5.3 AI 角色表现

#### 用户故事

作为用户，我希望角色的说话风格保持稳定、温柔、亲近，以便形成明确的人设感知。

#### 验收标准

1. 每次请求必须带 system prompt。
2. system prompt 必须约束角色为“圣代桥冰织”。
3. 回复风格应偏温柔、热恋、体贴，偶尔带一点害羞。
4. 回复末尾必须附带 emotion 标签，格式为 `[emotion:xxx]`。

### 5.4 emotion 标签与表情切换

#### 用户故事

作为用户，我希望冰织会根据对话内容切换表情，以便交互更生动。

#### 验收标准

1. 系统至少支持 5 种 emotion：`neutral`、`happy`、`shy`、`concerned`、`excited`。
2. AI 回复中的 emotion 标签必须被解析。
3. 若 emotion 对应资源存在，则桌宠切换到对应立绘。
4. 若 emotion 资源不存在，则保持当前立绘不变。
5. 对话结束一段时间后，应恢复为 `neutral`。

### 5.5 配置管理

#### 用户故事

作为用户，我希望能修改 AI 服务配置，以便连接自己的模型服务。

#### 验收标准

1. 系统首次启动时，若配置文件不存在，应自动生成默认配置。
2. 配置文件损坏时，应回退默认值并重建。
3. 配置项至少包括：
   - `llmEndpoint`
   - `llmApiKey`
   - `llmModel`
   - `ttsEndpoint`
   - `windowPos`
   - `voiceEnabled`
   - `volume`

### 5.6 AI 上下文管理

#### 用户故事

作为用户，我希望冰织记住最近的对话上下文，以便回复连续自然。

#### 验收标准

1. 系统应保留最近 10 轮上下文。
2. 每轮上下文由用户消息和助手消息组成。
3. 超过上限后，应自动裁剪最旧消息。

### 5.7 网络请求稳定性

#### 用户故事

作为用户，我希望网络临时异常时应用仍然足够稳定。

#### 验收标准

1. LLM 请求失败时，系统应自动重试。
2. 重试次数最多 3 次。
3. 重试间隔为 1 秒、2 秒、4 秒。
4. 若重试仍失败，应显示错误消息，不得导致程序崩溃。

## 6. 非功能需求

### 6.1 性能

- 主窗口启动时间应尽量控制在 3 秒内
- 所有网络请求必须异步执行
- 隐藏或空闲状态下不应有明显无意义占用

### 6.2 稳定性

- 配置文件异常不得导致程序直接崩溃
- 接口返回异常格式时应容错
- emotion 缺失时应回退 `neutral`

### 6.3 可维护性

- UI、配置、AI 会话、表情解析应分层实现
- 不应将 AI 拼包、JSON 解析、UI 更新逻辑全部堆在主窗口中

## 7. 后续扩展需求

以下需求不作为当前 MVP 的阻塞项，但需为后续留出扩展空间：

- `InputDialog`
- `SpeechBubble`
- `SettingsDialog`
- TTS 播报
- 系统托盘
- 单实例保护
- 日志滚动

## 8. 推荐默认配置

当前推荐的千问接入配置如下：

- `llmEndpoint`: `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`
- `llmModel`: `qwen-plus`

## 9. MVP 成功标准

当满足以下条件时，可视为当前阶段 MVP 完成：

- 启动时自动生成并读取配置
- 用户能输入文本并发送到真实 LLM
- 能收到真实 AI 回复
- 能解析 emotion 标签
- 能将 emotion 信号传回 UI
- 不因配置缺失或请求失败直接崩溃
